# Impact Analysis: spec #980 (tools/plan) on Checklist Dispatch Architecture

## What #980 Provides

A PEP 723 tool at `.opencode/tools/plan` wrapping `unified-planning` (v1.3.0). Accepts a YAML problem schema (objects, fluents, actions, preconditions, effects, init, goals) and generates an ordered action sequence via classical AI planning.

| Capability | What it does |
|-----------|-------------|
| `plan` | Generate action sequence from YAML problem definition |
| `validate` | Verify a plan achieves all goals — deterministic pass/fail |
| `ground` | Remove parameters, produce concrete executable actions |
| `state` | Shared state file management with `solve` (same format) |
| `discover` | List available planning engines |
| networkx integration | Cycle detection, topological sort, dependency DAG |

---

## Finding 1: #980 turns the self-generated tmp/ checklist from probabilistic to deterministic

**Current architecture (self-generated checklist)**: Sub-agent reads task file → LLM generates markdown checklist in tmp/ → plan quality gate (LLM-as-judge, probabilistic) → execute. Failure modes: hallucinated steps, wrong decomposition, verification confirmation bias.

**With #980**: Sub-agent reads task file → defines workflow as planning-problem YAML in tmp/ → runs `tools/plan` (deterministic planner) → validated plan written to tmp/ → execute. The plan is generated by classical AI planning, not by an LLM.

**Impact**: The 22 percentage-point improvement (72% → 94%) that Zylos attributes to explicit decomposition is a *lower bound* when the decomposition itself is deterministic. Classical planning with `unified-planning` eliminates:

| Failure Mode | LLM-generated checklist | `tools/plan` generated |
|-------------|------------------------|----------------------|
| Hallucinated prerequisite step | Common | Impossible (preconditions are formal) |
| Circular dependency | Common | Rejected (networkx cycle detection) |
| Wrong granularity | Common | Controlled (action definitions bound granularity) |
| Missing verification step | Common | Only if omitted from problem definition |
| Confirmation bias in verification | Common (same LLM checks own work) | Eliminated (deterministic validate) |

## Finding 2: #980 creates a three-layer verification stack

Current: `solve check` (Z3 invariants) + LLM plan quality gate (probabilistic).

With #980:

| Layer | Tool | What it verifies | Deterministic? |
|-------|------|-----------------|----------------|
| Plan generation | `tools/plan` | Action sequence achieves goal state | ✅ Deterministic (classical planning) |
| Plan validation | `tools/plan validate` | Plan achieves all goals, no cycles | ✅ Deterministic |
| Invariant checking | `tools/solve check` | State invariants hold across all steps | ✅ Deterministic (Z3) |

All three layers are now deterministic. No LLM-as-judge needed for plan quality or plan adherence. The only probabilistic step is the sub-agent's interpretation of the task file into a planning-problem YAML — which is a much simpler cognitive task than generating the full execution sequence.

## Finding 3: #980 changes what goes in tmp/ during sub-agent execution

Before:
```markdown
# tmp/checklist-{task}.md
- [ ] 1. Load config file
- [ ] 2. Run pre-processing
- [ ] 3. Execute transform
- [ ] 4. Verify output
```

After:
```yaml
# tmp/plan-problem-{task}.yaml
domain: implementation-pipeline
objects:
  - name: spec_file
    type: artifact
  - name: branch
    type: git_state
fluents:
  - name: spec_approved
    type: bool
    params: [{name: s, type: artifact}]
  - name: branch_created
    type: bool
    params: [{name: b, type: git_state}]
actions:
  - name: create_branch
    params: [{name: b, type: git_state}]
    preconditions: ["spec_approved(spec_file)"]
    effects: ["branch_created(b)", "not spec_approved(spec_file)"]
init:
  - "spec_approved(spec_file)"
goals:
  - "branch_created(branch)"
```

Then: `tools/plan --problem tmp/plan-problem-{task}.yaml` → deterministic action sequence.

## Finding 4: #980 changes the skill card checklist format (optional upgrade path)

The SKILL.md checklist row currently specifies a sub-agent dispatch. With #980, the checklist row could optionally include the planning-problem domain for the sub-agent to use:

```
- [ ] 1. (blind) task(subagent_type="general", prompt="execute pre-work from git-workflow")
                 domain: git-workflow  # optional — sub-agent uses tools/plan internally
```

This doesn't change the orchestrator's dispatch behavior. It's metadata the sub-agent consumes to generate its own plan via `tools/plan`. The orchestrator still dispatches blind.

## Finding 5: #980 + `solve` together enable full workflow verification

The composition pattern from #980's spec (`solve check` validates plan invariants via Z3; `plan plan` generates the action sequence) maps directly onto our architecture:

```
1. Sub-agent loads task file
2. Sub-agent defines planning-problem YAML in tmp/
3. Sub-agent runs: tools/plan plan --problem tmp/problem.yaml > tmp/plan.yaml
4. Sub-agent runs: tools/solve check --contract tmp/contract.yaml --state tmp/state.yaml
5. If both PASS: execute against tmp/plan.yaml
6. If either FAIL: regenerate problem definition (not hallucinate — refine the formal spec)
```

This creates a **deterministic gate loop** — not a probabilistic LLM-as-judge loop.

## Finding 6: Where #980 does NOT apply

Classical planning requires formalized actions with known preconditions and effects. It applies to:

| Work | Applies? | Why |
|------|----------|-----|
| Implementation pipeline (pre-work → implement → verify → audit → commit → PR) | ✅ | Steps are known, preconditions/effects are enumerable |
| Git workflow (branch → commit → push → PR → merge) | ✅ | Deterministic state machine |
| Verification pipeline (RED → GREEN → VbC → audit) | ✅ | Well-defined gates with pass/fail conditions |
| Open-ended research | ❌ | Actions unknown in advance |
| Brainstorming | ❌ | Exploration, not execution |
| Bug investigation | ❌ | Unknown root cause, emergent path |

The hybrid pattern remains: pre-authored SKILL.md checklist for orchestrator dispatch (always), plus either:
- `tools/plan` deterministic decomposition (for formalizable work by sub-agents)
- LLM-generated tmp/ checklist (for open-ended work by sub-agents)

## Finding 7: Dependency on #980

#980 does NOT depend on #1010 (it's a standalone tool). But #1010 benefits from #980 because:

1. **Sub-agent plan quality becomes deterministic** — eliminates the plan verification gap that Zylos identifies (same-LLM confirmation bias)
2. **tmp/ checklists become executable artifacts** — not markdown to follow, but YAML plans a deterministic engine validates
3. **Full three-layer verification stack** — `tools/plan` (generation) + `tools/plan validate` (correctness) + `tools/solve check` (invariants) — all deterministic
4. **Plan caching benefit** — planning-problem YAMLs can be cached per workflow type (like the pre-authored SKILL.md checklist), with invalidation when action definitions change

**Recommendation**: #980 should be implemented before or in parallel with Phase 3 (behavioral enforcement) of #1010. It doesn't block Phases 1 or 2 (pre-read cascade + checklist format), but it significantly improves the sub-agent layer once sub-agents start generating self-checklists.

## Summary

| Aspect | Without #980 | With #980 |
|--------|-------------|-----------|
| Sub-agent plan generation | LLM (probabilistic) | `tools/plan` (deterministic) |
| Plan quality gate | LLM-as-judge (confirmation bias risk) | `plan validate` + `solve check` (deterministic) |
| tmp/ artifact format | Markdown checklist | YAML planning-problem + action sequence |
| Verification stack | Single layer (solve) | Three layers (plan → validate → check) |
| Cycle detection | LLM may miss it | networkx detects automatically |
| Hallucinated steps | Common failure mode | Impossible (formal preconditions) |
| Orchestrator dispatch behavior | Unchanged | Unchanged (orchestrator stays blind) |
