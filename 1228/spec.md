# [SPEC-FIX] Mandate compliance statement in every spec and plan body

## Problem

AI agents implementing specs and executing plans will, when faced with multiple verification gates and procedural steps, rationalize that "too many steps" justifies skipping or compressing required work. This produces downstream defects that are only discovered when the feature branch is rejected — requiring a full rework from scratch.

The root cause is the absence of an explicit consequence statement in the artifacts themselves. Agents read the spec/plan body and see only the work description. Without a statement that skipping steps = branch rejection + total rework, the agent has no explicit artifact-level signal that step-skipping is unacceptable.

## Proposed Fix

Add a fixed, identical compliance statement to every spec and plan generated by `spec-creation` and `writing-plans`. The statement must appear in two locations within the body:

1. **Top** (after the preamble/executive summary, before the main body content)
2. **Bottom** (before the success criteria / exit criteria section)

### Wording (exact, word-for-word)

> **Compliance Requirement:** All steps and sub-steps in this document MUST be followed in order. Failure to comply with any step — including but not limited to verification gates, test phases, audit checkpoints, and review steps — will result in the feature branch being rejected and discarded, requiring a full rework from scratch and loss of all prior work. There is no valid reason to skip, compress, reorder, or omit any step. If a step appears redundant or unnecessary, follow it anyway — the cost of following an extra step is negligible compared to the cost of rework from a skipped step.

### Where It Lives

The mandate is enforced at artifact generation time in two skill task files:

| Artifact | File to Modify | Insertion Point |
|----------|---------------|-----------------|
| Spec body | `spec-creation/tasks/write.md` | Step 1 (Assemble Spec) — mandate that the spec body includes the statement at top and bottom |
| Plan body | `writing-plans/tasks/create/create-and-validate.md` | Step 7 (Store Plan Document) — mandate that the plan body includes the statement at top and bottom |

### Enforcement Tests

Two behavioral enforcement tests verify that generated artifacts carry the statement.

#### Behavioral Test 1: Spec Contains Compliance Statement

- **RED:** Generate a spec using `spec-creation`. Verify the spec body does NOT currently contain the compliance statement (because the task file hasn't been updated yet).
- **GREEN:** After modifying `spec-creation/tasks/write.md`, re-generate the spec. Verify the spec body contains the compliance statement at top and bottom.

#### Behavioral Test 2: Plan Contains Compliance Statement

- **RED:** Generate a plan using `writing-plans`. Verify the plan body does NOT currently contain the compliance statement.
- **GREEN:** After modifying `writing-plans/tasks/create/create-and-validate.md`, re-generate the plan. Verify the plan body contains the compliance statement at top and bottom.

### Content-Verification Test

Grep both modified task files for the compliance statement mandate text:

```bash
grep -c "Compliance Requirement" .opencode/skills/spec-creation/tasks/write.md
grep -c "Compliance Requirement" .opencode/skills/writing-plans/tasks/create/create-and-validate.md
```

Both must return at least 1 (the mandate reference in the task procedure).

## Scope

**In scope:**
- Modifying `spec-creation/tasks/write.md` Step 1 to mandate the compliance statement in spec bodies
- Modifying `writing-plans/tasks/create/create-and-validate.md` Step 7 to mandate the compliance statement in plan bodies
- Behavioral enforcement test in `.opencode/tests/behaviors/`
- Content-verification test update

**Out of scope:**
- Modifying the existing spec templates or plan templates beyond adding the mandate
- Adding the statement retroactively to existing artifacts
- Non-generated artifacts (hand-written specs)
